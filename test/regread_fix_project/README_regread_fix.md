# 寄存器显示调试口修正版说明

本文件夹基于你上传的 Vivado 拆分工程文件修改，目标是让板子支持：

```text
SW[5]   = 1
SW[4:0] = 寄存器编号
```

通过数码管查看对应通用寄存器 `x0 ~ x31` 的当前值。

---

## 1. 原来为什么看 x14 还是 00000000？

你原工程中的 `Multi_CH32.v` 已经有如下选择逻辑：

```verilog
6'b1xxxxx: seg7_data = reg_data;
```

这说明当 `SW[5]=1` 时，数码管确实会切到 `reg_data` 通道。

但是原来的 `RVCPUSOC_Top.v` 中同时写了：

```verilog
assign reg_data = 32'b0;
```

因此 `reg_data` 被硬连为 0。无论你选择 `x14`、`x30` 还是其他寄存器，显示都会是：

```text
00000000
```

这不是寄存器真实值为 0，而是寄存器调试通道没有真正接到寄存器堆。

---

## 2. 本次修改内容

### 2.1 修改 `parts.sv`

给 `regfile` 增加第三个调试读口：

```verilog
input  logic [4:0]  a4,
output logic [31:0] rd4
```

并增加：

```verilog
assign rd4 = (a4 != 0) ? rf[a4] : 32'b0;
```

这样寄存器堆除了 CPU 正常使用的 `rs1/rs2` 两个读口外，还能额外根据 `SW[4:0]` 读出任意寄存器。

---

### 2.2 修改 `riscvEnd.sv`

给 `riscvpipeline` 顶层增加两个端口：

```verilog
input  logic [4:0]  reg_sel,
output logic [31:0] reg_data
```

并将它们接入 `regfile`：

```verilog
regfile rf(clk, RegWriteW, InstrD[19:15], InstrD[24:20],
           rdW, reg_sel, ResultW, ReadData1D, ReadData2D, reg_data);
```

---

### 2.3 修改 `RVCPUSOC_Top.v`

删除原来的：

```verilog
assign reg_data = 32'b0;
```

并在 CPU 实例化时加入：

```verilog
.reg_sel(sw_i[4:0]),
.reg_data(reg_data),
```

这样完整链路变成：

```text
SW[4:0]
  -> reg_sel
  -> regfile 第三读口
  -> reg_data
  -> Multi_CH32
  -> 数码管显示
```

## 3. 如何使用 SW 查看寄存器？

查看寄存器时，令：

```text
SW[5] = 1
SW[4:0] = 寄存器编号
```

也就是：

```text
SW[5:0] = 1_xxxxx
```

其中 `xxxxx` 是 5 位寄存器号。

常见例子：

| 寄存器 | 编号二进制 | SW[5:0] | 说明 |
|---|---:|---:|---|
| x0  | 00000 | 100000 | 恒为 0 |
| x1  | 00001 | 100001 | 返回地址 ra |
| x2  | 00010 | 100010 | 程序中常作 switch IO 地址 |
| x3  | 00011 | 100011 | 程序中常作 seg7 IO 地址 |
| x10 | 01010 | 101010 | 显示值/结果寄存器 |
| x14 | 01110 | 101110 | 取决于当前汇编程序用途 |
| x20 | 10100 | 110100 | Feistel 左半块 L |
| x21 | 10101 | 110101 | Feistel 右半块 R |
| x24 | 11000 | 111000 | K0 |
| x25 | 11001 | 111001 | K1 |
| x26 | 11010 | 111010 | K2 |
| x27 | 11011 | 111011 | K3 |
| x30 | 11110 | 111110 | 0x0000FFFF mask |
| x31 | 11111 | 111111 | 子程序返回地址暂存 |

例如查看 `x14`：

```text
SW[5:0] = 101110
```

查看 `x30`：

```text
SW[5:0] = 111110
```

## 4. 为什么有些寄存器仍可能显示 00000000？

修正调试口后，如果某个寄存器显示 0，可能有三种原因：

1. 该寄存器确实还没有被程序写入。
2. 程序已经执行到后面，那个寄存器被新值覆盖成 0。
3. CPU 仍在高速运行，寄存器值正在变化，人眼看到的是某一瞬间值。

建议调试时可以：

```text
SW15 = 1
```

让 CPU 使用较慢时钟，便于观察。

如果是查看最终结果，建议等程序进入最终死循环后再查看，此时大部分寄存器会稳定。

## 5. 如何逐步检查寄存器调试功能是否正常？

### 第一步：检查显示通道

拨：

```text
SW[5:0] = 100000
```

应该显示：

```text
00000000
```

因为 `x0` 永远是 0。

### 第二步：查看 IO 地址寄存器

如果你的程序开头设置了：

```asm
x2 = 0xffff0004
x3 = 0xffff000c
```

那么拨：

```text
SW[5:0] = 100010   # x2
```

应能看到：

```text
FFFF0004
```

拨：

```text
SW[5:0] = 100011   # x3
```

应能看到：

```text
FFFF000C
```

如果这两个能显示，说明寄存器调试口已经接通。

### 第三步：查看关键算法寄存器

以 Feistel 程序为例：

```text
SW[5:0] = 111110   # x30，应为 0000FFFF
SW[5:0] = 111000   # x24，K0
SW[5:0] = 111001   # x25，K1
SW[5:0] = 111010   # x26，K2
SW[5:0] = 111011   # x27，K3
SW[5:0] = 101010   # x10，当前显示值/最终结果
```

---

## 6. 注意事项

1. `SW[5]=1` 时，`SW[4:0]` 用作寄存器编号，因此此时不能同时把 `SW[4:0]` 用作应用输入。
2. 正常应用显示仍使用：

```text
SW[5:0] = 000000
```

3. 查看 PC、指令、访存地址等调试通道仍使用：

```text
SW[5:0] = 000010   # PC
SW[5:0] = 000011   # instr
SW[5:0] = 000100   # cpu_data_addr
SW[5:0] = 000101   # cpu_data_out
SW[5:0] = 000110   # dm_dout
SW[5:0] = 000111   # ram_addr converted to byte address
```
