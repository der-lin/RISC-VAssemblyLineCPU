# easy_cry_interactive：交互式 Feistel 加密/解密工程

## 1. 工程目标

本版本支持两种模式：

```text
SW14 = 0：加密模式，输入明文和密钥，输出密文
SW14 = 1：解密模式，输入密文和密钥，输出明文
```

输入通过拨码开关完成：

```text
SW[9:6] = 当前输入的 1 位十六进制数字，范围 0~F
SW13    = 确认当前这一位；确认后需要拨回 0
SW14    = 模式选择；建议在复位前设置好
SW[5:0] = 000000 时显示 CPU 程序输出
```

## 2. 使用流程

### 加密模式

1. 设 `SW14 = 0`。
2. 设 `SW[5:0] = 000000`。
3. 复位 CPU。
4. 板子先显示 `E0000000`，表示加密模式。
5. 使用 `SW[9:6] + SW13` 输入 8 位明文，例如 `12345678`。
6. 板子显示 `A0000000`，表示开始输入 key。
7. 使用 `SW[9:6] + SW13` 输入 4 位 key，例如 `2B7E`。
8. 程序输出并保持显示密文。若输入 `plaintext=12345678, key=2B7E`，应显示：

```text
36C8930C
```

### 解密模式

1. 设 `SW14 = 1`。
2. 设 `SW[5:0] = 000000`。
3. 复位 CPU。
4. 板子先显示 `D0000000`，表示解密模式。
5. 使用 `SW[9:6] + SW13` 输入 8 位密文，例如 `36C8930C`。
6. 板子显示 `A0000000`，表示开始输入 key。
7. 使用 `SW[9:6] + SW13` 输入 4 位 key，例如 `2B7E`。
8. 程序输出并保持显示明文。若输入 `ciphertext=36C8930C, key=2B7E`，应显示：

```text
12345678
```

## 3. Feistel 算法

32-bit 分组拆成两个 16-bit 半块：

```text
L = block[31:16]
R = block[15:0]
```

轮函数：

```text
F(R,K) = ((R xor K) + ((R << 3) xor (R >> 2))) & 0xFFFF
```

加密轮：

```text
L' = R
R' = L xor F(R,K)
```

解密轮使用反向密钥，并执行逆轮：

```text
oldR = L'
oldL = R' xor F(L',K)
```

轮密钥由 16-bit master key 派生：

```text
K0 = key
K1 = ((key << 3) xor 0x1357) & 0xFFFF
K2 = ((key >> 1) xor 0x2468) & 0xFFFF
K3 = ((key << 5) xor (key >> 3) xor 0x55AA) & 0xFFFF
```

本算法仅用于教学演示和 CPU 功能验证，不作为真实安全密码算法使用。

|---|---|---|

## 5. Vivado 使用说明

1. 将本文件夹中的 HDL 文件加入工程：
   - `RVCPUSOC_Top.v`
   - `riscvEnd.sv`
   - `parts.sv`
   - `dm.v`
   - `MIO_BUS.v`
   - `Multi_CH32.v`
   - `seg7x16.v`
   - `clk_div.v`
   - `Nexys4DDR_CPU.xdc`
2. 指令 ROM IP：
   - `Data Width = 32`
   - `Depth = 512`
   - 初始化文件选择 `easy_cry_interactive.coe`
3. 数据 RAM 已在 `dm.v` 中扩展为 256 word：
   - `reg [31:0] RAM[255:0]`
   - `ram_addr = cpu_data_addr[9:2]`
4. 建议操作时：
   - 每确认一位后，将 `SW13` 拨回 0。
   - 模式 `SW14` 建议在复位前设置好，程序启动后只读取一次。

## 7. 本版本新增：`SW[5]=1` 查看寄存器

### 7.1 结论

本版本已经接通寄存器调试读口，可以通过：

```text
SW[5]   = 1
SW[4:0] = 寄存器编号
```

查看寄存器堆中对应寄存器的值。也就是说，`SW[5:0] = 1xxxxx` 时，`Multi_CH32` 会选择 `reg_data` 通道，而 `reg_data` 现在已经接到 CPU 寄存器堆的调试读口。

例如：

| 想查看的寄存器 | `SW[4:0]` | `SW[5:0]` | 说明 |
|---|---:|---:|---|
| `x0` | `00000` | `100000` | 恒为 `00000000` |
| `x1` | `00001` | `100001` | 返回地址寄存器 |
| `x2` | `00010` | `100010` | 开关输入地址，通常为 `FFFF0004` |
| `x3` | `00011` | `100011` | 数码管输出地址，通常为 `FFFF000C` |
| `x10` | `01010` | `101010` | 显示值 / 结果值 |
| `x11` | `01011` | `101011` | 输入累计值 / key 暂存 |
| `x20` | `10100` | `110100` | Feistel 左半块 `L` |
| `x21` | `10101` | `110101` | Feistel 右半块 `R` |
| `x22` | `10110` | `110110` | 当前轮密钥 |
| `x24` | `11000` | `111000` | `K0` |
| `x25` | `11001` | `111001` | `K1` |
| `x26` | `11010` | `111010` | `K2` |
| `x27` | `11011` | `111011` | `K3` |
| `x30` | `11110` | `111110` | `0x0000FFFF` 掩码 |
| `x31` | `11111` | `111111` | `encrypt32/decrypt32` 中保存的返回地址 |

因此，若程序已经执行过初始化阶段，查看 `x30` 时应显示：

```text
0000FFFF
```

若显示仍为 `00000000`，优先检查是否下载了本版本的 bitstream，或者是否仍在使用旧版 `RVCPUSOC_Top.v / riscvEnd.sv / parts.sv`。

### 7.2 本版本改了哪些硬件连接

相较于旧版，本版本做了三处 HDL 修改：

1. `parts.sv` 中的 `regfile` 增加第三个调试读口：

```verilog
input  logic [4:0]  a4,
output logic [31:0] rd4
```

其中 `a4` 是寄存器编号，`rd4` 是该寄存器的值。

2. `riscvEnd.sv` 中的 `riscvpipeline` 增加端口：

```verilog
input  logic [4:0]  reg_sel,
output logic [31:0] reg_data
```

并将 `reg_sel` 传给 `regfile` 的第三读口。

3. `RVCPUSOC_Top.v` 中将 `sw_i[4:0]` 接到 CPU：

```verilog
.reg_sel(sw_i[4:0]),
.reg_data(reg_data),
```

同时删除旧版中的：

```verilog
assign reg_data = 32'b0;
```

所以现在 `SW[5]=1` 不再只是形式上选择 `reg_data`，而是真的可以查看寄存器堆中的值。

### 7.3 查看寄存器时的注意事项

1. 程序快速运行时，寄存器值会变化很快。建议先让程序进入最终结果循环，再查看寄存器。
2. 如果想观察执行过程中的寄存器变化，可以设置 `SW15=1` 使用慢速 CPU 时钟。
3. 查看寄存器时，`SW[5:0]` 不再是 `000000`，所以数码管不会显示应用输出，而是显示寄存器值。
4. 应用程序本身只使用 `SW14`、`SW13`、`SW[9:6]` 作为交互输入，`SW[5:0]` 主要用于显示通道选择；查看寄存器不会改变加密/解密算法本身。

## 8. 如果程序出问题，如何逐层检查

建议按“从硬件显示到程序逻辑”的顺序检查，不要一开始就怀疑加密算法。

### 8.1 第 1 关：确认指令 ROM 和启动模式

设置：

```text
SW[5:0] = 000000
SW14 = 0    # 加密模式
```

复位后应先显示：

```text
E0000000
```

如果设置 `SW14=1` 后复位，应显示：

```text
D0000000
```

若没有出现 `E0000000/D0000000`，优先检查：

1. 指令 ROM IP 是否加载了 `easy_cry_interactive.coe`；
2. 指令 ROM `Depth` 是否设为 `512`，`Data Width` 是否为 `32`；
3. 顶层中 `imem` 地址是否为 `pc[10:2]`；
4. 复位信号是否正常。

### 8.2 第 2 关：确认输入逻辑

保持：

```text
SW[5:0] = 000000
```

输入明文 `12345678` 时，数码管应逐步显示：

```text
00000001
00000012
00000123
00001234
00012345
00123456
01234567
12345678
```

输入 key `2B7E` 时，应逐步显示：

```text
A0000002
A000002B
A00002B7
A0002B7E
```

如果输入不变化，检查：

1. 是否使用 `SW[9:6]` 输入当前 1 位十六进制数字；
2. 每次确认后是否把 `SW13` 拨回 0；
3. 是否误把输入数字拨在 `SW[3:0]` 上；
4. `MIO_BUS.v` 中读取开关地址 `0xFFFF0004` 是否正确返回 `{16'h0, sw_i}`。

### 8.3 第 3 关：确认数据写入 RAM

可以通过调试通道查看访存地址和写数据：

| `SW[5:0]` | 显示内容 | 用途 |
|---|---|---|
| `000100` | `cpu_data_addr` | 当前数据地址 / ALU 结果 |
| `000101` | `cpu_data_out` | 当前写出数据 |
| `000110` | `dm_dout` | 数据 RAM 读出值 |
| `000111` | `{ram_addr, 2'b00}` | RAM 地址还原后的字节地址 |

加密模式下，程序应把关键数据写入：

| 地址 | 预期内容，以 plaintext=`12345678`, key=`2B7E` 为例 |
|---|---|
| `0x180` | `12345678` |
| `0x184` | `00002B7E` |
| `0x188` | `36C8930C` |
| `0x194` | `36C8930C` |
| `0x198` | `00002B7E` |
| `0x19C` | `000048A7` |
| `0x1A0` | `000031D7` |
| `0x1A4` | `00003F05` |

如果地址 `0x180~0x1A4` 不正常，检查数据 RAM 地址扩展：

```verilog
ram_addr = cpu_data_addr[9:2]
```

并确认 `dm.v` 中 RAM 为：

```verilog
reg [31:0] RAM[255:0];
```

### 8.4 第 4 关：确认 PC 和当前指令

| `SW[5:0]` | 显示内容 |
|---|---|
| `000001` | `PC / 4`，即指令编号 |
| `000010` | `PC` |
| `000011` | 当前指令机器码 `instr` |

如果程序输入后没有继续运行，可以切到：

```text
SW[5:0] = 000010
```

观察 PC 是否停在某个等待循环。

常见现象：

1. PC 停在输入循环附近：说明程序还在等待 `SW13=1` 或等待 `SW13=0`。
2. PC 进入最后自循环：说明程序已经算完并保持结果。
3. PC 回到开头：检查复位信号或 bitstream 是否异常。

### 8.5 第 5 关：确认关键寄存器

本版本可通过 `SW[5]=1` 查看寄存器。

推荐检查：

| 寄存器 | 期望值或含义 |
|---|---|
| `x2` | `FFFF0004`，开关输入地址 |
| `x3` | `FFFF000C`，数码管输出地址 |
| `x30` | `0000FFFF`，16-bit 掩码 |
| `x24` | `K0` |
| `x25` | `K1` |
| `x26` | `K2` |
| `x27` | `K3` |
| `x10` | 最终结果或显示值 |

以 `key=2B7E` 为例，轮密钥应为：

```text
x24 = 00002B7E
x25 = 000048A7
x26 = 000031D7
x27 = 00003F05
```

加密完成后，`x10` 通常应保持最终密文：

```text
36C8930C
```

1. 设置：

```text
SW14 = 0
SW[5:0] = 000000
```

2. 复位，板子显示：

```text
E0000000
```

3. 通过 `SW[9:6] + SW13` 输入明文：

```text

```

4. 板子显示：

```text
A0000000
```

```text

```

6. 最终应显示密文：

```text
36C8930C
```

可以向老师说明：程序已完成“输入明文 → 输入密钥 → 轮密钥派生 → 4 轮 Feistel 加密 → 输出密文”。

### 9.3 解密演示流程

1. 设置：

```text
SW14 = 1
SW[5:0] = 000000
```

2. 复位，板子显示：

```text
D0000000
```

3. 输入密文：

```text
36C8930C
```

4. 输入 key：

```text
2B7E
```

5. 最终应显示明文：

```text
12345678
```

可以向老师说明：解密过程使用相同轮函数，但轮密钥顺序反向，从而恢复原明文。

### 9.4 调试演示流程

如果老师想看程序运行过程，可以按以下顺序展示：

1. `SW[5:0]=000010`：查看 PC，说明程序正在执行而不是静态显示。
2. `SW[5:0]=000011`：查看当前机器码，说明 CPU 从指令 ROM 取指执行。
3. `SW[5:0]=000100`：查看数据地址，例如 `0x180/0x184/0x188/0x194/0x198~0x1A4`。
4. `SW[5:0]=000101`：查看写入数据，例如明文、key、轮密钥、密文。
5. `SW[5]=1` 且 `SW[4:0]=11110`：查看 `x30=0000FFFF`。
6. `SW[5]=1` 且 `SW[4:0]=11000~11011`：查看 `x24~x27` 中的轮密钥。

### 9.5 推荐答辩表述

可以这样介绍：

> 本程序是一个基于简化 Feistel 结构的交互式加解密程序。CPU 通过 `0xFFFF0004` 读取拨码开关，通过 `0xFFFF000C` 输出到数码管。程序支持加密和解密两种模式，使用 `SW14` 选择模式，使用 `SW[9:6]` 输入一位十六进制数，使用 `SW13` 确认。加密时，输入明文 `0x12345678` 和 key `0x2B7E`，程序派生 4 个轮密钥并执行 4 轮 Feistel 加密，最终输出 `0x36C8930C`。解密时输入该密文和同一 key，程序使用反向轮密钥恢复明文 `0x12345678`。同时，本版本已经接通寄存器调试读口，可通过 `SW[5]=1` 和 `SW[4:0]` 查看任意寄存器值，例如 `x30=0x0000FFFF` 掩码和 `x24~x27` 轮密钥。

## 10. 常见问题

### Q1：为什么输入数字时不能用 `SW[3:0]`？

因为 `SW[5:0]` 被 `Multi_CH32` 用作显示通道选择。如果使用 `SW[3:0]` 输入数字，数码管显示通道会跟着变化。为避免冲突，本程序使用 `SW[9:6]` 输入当前十六进制数字。

### Q2：为什么 `SW14` 要在复位前设置好？

程序只在启动阶段读取一次 `SW14` 来选择加密或解密模式。运行过程中切换 `SW14` 不会改变当前模式。如果要切换模式，需要重新设置 `SW14` 后复位。

### Q3：为什么有时输入没有反应？

通常是因为 `SW13` 没有拨回 0。每确认一位后，程序会等待 `SW13=0`，然后才接受下一位输入。

### Q4：查看寄存器时为什么应用输出消失？

因为 `SW[5]=1` 时显示通道切换到 `reg_data`。要回到程序输出显示，请将 `SW[5:0]` 改回 `000000`。

### Q5：如果 `x30` 不是 `0000FFFF` 怎么办？

先确认使用的是本版本 bitstream，然后检查：

1. `riscvEnd.sv` 是否包含 `reg_sel/reg_data`；
2. `parts.sv` 的 `regfile` 是否包含第三读口；
3. `RVCPUSOC_Top.v` 是否连接 `.reg_sel(sw_i[4:0])` 和 `.reg_data(reg_data)`；
4. 是否仍然存在 `assign reg_data = 32'b0;`。本版本中该语句应已删除。
