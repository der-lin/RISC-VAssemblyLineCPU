# easy_cry_interactive：交互式 Feistel 加密/解密工程

## 1. 工程目标

本版本支持两种模式：

```text
SW14 = 0：加密模式，输入明文和密钥，输出密文
SW14 = 1：解密模式，输入密文和密钥，输出明文
```

输入通过拨码开关完成：

```text
SW[9:6] = 当前输入的 1 位十六进制数字，范围 0~F
SW13    = 确认当前这一位；确认后需要拨回 0
SW14    = 模式选择；建议在复位前设置好
SW[5:0] = 000000 时显示 CPU 程序输出
```

## 2. 使用流程

### 加密模式

1. 设 `SW14 = 0`。
2. 设 `SW[5:0] = 000000`。
3. 复位 CPU。
4. 板子先显示 `E0000000`，表示加密模式。
5. 使用 `SW[9:6] + SW13` 输入 8 位明文，例如 `12345678`。
6. 板子显示 `A0000000`，表示开始输入 key。
7. 使用 `SW[9:6] + SW13` 输入 4 位 key，例如 `2B7E`。
8. 程序输出并保持显示密文。若输入 `plaintext=12345678, key=2B7E`，应显示：

```text
36C8930C
```

### 解密模式

1. 设 `SW14 = 1`。
2. 设 `SW[5:0] = 000000`。
3. 复位 CPU。
4. 板子先显示 `D0000000`，表示解密模式。
5. 使用 `SW[9:6] + SW13` 输入 8 位密文，例如 `36C8930C`。
6. 板子显示 `A0000000`，表示开始输入 key。
7. 使用 `SW[9:6] + SW13` 输入 4 位 key，例如 `2B7E`。
8. 程序输出并保持显示明文。若输入 `ciphertext=36C8930C, key=2B7E`，应显示：

```text
12345678
```

## 3. Feistel 算法

32-bit 分组拆成两个 16-bit 半块：

```text
L = block[31:16]
R = block[15:0]
```

轮函数：

```text
F(R,K) = ((R xor K) + ((R << 3) xor (R >> 2))) & 0xFFFF
```

加密轮：

```text
L' = R
R' = L xor F(R,K)
```

解密轮使用反向密钥，并执行逆轮：

```text
oldR = L'
oldL = R' xor F(L',K)
```

轮密钥由 16-bit master key 派生：

```text
K0 = key
K1 = ((key << 3) xor 0x1357) & 0xFFFF
K2 = ((key >> 1) xor 0x2468) & 0xFFFF
K3 = ((key << 5) xor (key >> 3) xor 0x55AA) & 0xFFFF
```

本算法仅用于教学演示和 CPU 功能验证，不作为真实安全密码算法使用。

## 7. 关于 `SW[5]=1` 查看寄存器功能的检查结论

### 7.1 当前版本结论

当前 `easy_cry_interactive` 这一版 **不能通过 `SW[5]=1` + `SW[4:0]` 正确查看寄存器堆中的真实寄存器值**。

原因不是 `Multi_CH32.v` 没有选择寄存器通道，而是寄存器数据通路没有接通。当前文件中存在以下事实：

1. `Multi_CH32.v` 中确实保留了寄存器显示选择逻辑：

```verilog
6'b1xxxxx: seg7_data = reg_data;
```

这表示当 `SW[5]=1` 时，数码管显示的是 `reg_data` 这根线。

2. 但是在 `RVCPUSOC_Top.v` 中，`reg_data` 被直接固定为 0：

```verilog
// riscvpipeline in riscvEnd.sv has no debug register readout port
assign reg_data = 32'b0;
```

因此只要进入 `SW[5]=1` 的寄存器显示通道，实际显示的都是 `00000000`。

3. 当前 `riscvEnd.sv` 的 `riscvpipeline` 顶层端口没有 `reg_sel` / `reg_data` 这类调试寄存器读口。

4. 当前 `parts.sv` 中的 `regfile` 只有正常 CPU 执行需要的两个读口 `rd1/rd2`，没有额外的第三个调试读口，因此 `SW[4:0]` 并没有真正接入寄存器堆。

所以，如果你拨：

```text
SW[5]   = 1
SW[4:0] = 11110    # 期望查看 x30
```

看到：

```text
00000000
```

并不说明 `x30` 真的等于 0。它只说明当前版本中 `reg_data` 没有接通，而是被硬连成了 0。

### 7.2 如何验证 `x30 = 0x0000FFFF` 是否真的写入？

`easy_cry_interactive.asm` 中使用 `x30` 保存 16-bit 掩码：

```asm
li    x30, 0x0000ffff
```

如果想验证 `x30` 是否正确，可以临时在这条指令后增加：

```asm
addi  x10, x30, 0
sw    x10, 0(x3)
```

然后保持：

```text
SW[5:0] = 000000
```

如果数码管显示：

```text
0000FFFF
```

就说明 `x30` 已经被正确写入。这个方法比 `SW[5]=1` 查看寄存器更可靠，因为它直接让程序把 `x30` 写到数码管 IO 地址 `0xffff000c`。

### 7.3 如果后续想恢复 `SW[5]=1` 查看寄存器

需要补齐以下链路：

```text
SW[4:0]
  -> reg_sel
  -> regfile 调试读口
  -> reg_data
  -> Multi_CH32
  -> 数码管
```

也就是说，需要：

1. 在 `regfile` 中增加第三读口；
2. 在 `riscvpipeline` 顶层增加 `reg_sel` 和 `reg_data` 端口；
3. 在 `RVCPUSOC_Top.v` 中把 `sw_i[4:0]` 接入 `reg_sel`；
4. 删除 `assign reg_data = 32'b0;`；
5. 保留 `Multi_CH32.v` 中的 `6'b1xxxxx: seg7_data = reg_data;`。

示意代码如下。

`parts.sv` 中的 `regfile` 可改为：

```verilog
module regfile(
    input  logic        clk,
    input  logic        we3,
    input  logic [4:0]  a1, a2, a3,
    input  logic [4:0]  a4,
    input  logic [31:0] wd3,
    output logic [31:0] rd1, rd2,
    output logic [31:0] rd4
);

  logic [31:0] rf[31:0];

  always_ff @(negedge clk)
    if (we3 && (a3 != 0))
      rf[a3] <= wd3;

  assign rd1 = (a1 != 0) ? rf[a1] : 32'b0;
  assign rd2 = (a2 != 0) ? rf[a2] : 32'b0;
  assign rd4 = (a4 != 0) ? rf[a4] : 32'b0;

endmodule
```

`riscvEnd.sv` 的 `riscvpipeline` 端口需要增加：

```verilog
input  logic [4:0]  reg_sel,
output logic [31:0] reg_data,
```

并在实例化 `regfile` 时接入第三读口。

`RVCPUSOC_Top.v` 中 CPU 实例需要增加：

```verilog
.reg_sel(sw_i[4:0]),
.reg_data(reg_data),
```

并删除：

```verilog
assign reg_data = 32'b0;
```

## 8. 如果程序出问题，如何一个关节一个关节检查

调试时不要一上来怀疑整个 CPU。建议按下面顺序逐层定位。

### 8.1 第 0 关：确认显示通道

操作：

```text
SW[5:0] = 000000
```

含义：显示 CPU 程序写入 `0xffff000c` 的应用输出。

如果 `SW[5:0]` 不是 `000000`，数码管可能显示 PC、指令、数据地址、RAM 输出等调试通道，而不是应用程序输出。

### 8.2 第 1 关：确认模式选择是否正确

复位前设置：

```text
SW14 = 0 -> 加密模式
SW14 = 1 -> 解密模式
```

复位后观察：

```text
E0000000 -> 进入加密模式
D0000000 -> 进入解密模式
```

如果模式显示不对，重点检查：

1. `SW14` 是否在复位前已经设置好；
2. `MIO_BUS.v` 是否将 `0xffff0004` 映射到 `{16'h0, sw_i}`；
3. 程序是否正确读取 `SW14`：

```asm
lw    x5, 0(x2)
srli  x6, x5, 14
andi  x6, x6, 1
```

### 8.3 第 2 关：确认十六进制输入是否有效

输入数字使用：

```text
SW[9:6] = 当前十六进制位
SW13    = 确认当前位
```

每次确认都要完成：

```text
SW13: 0 -> 1 -> 0
```

输入过程中，数码管应显示当前累计值。例如加密模式输入明文 `12345678` 时，显示会逐步变化：

```text
00000001
00000012
00000123
00001234
00012345
00123456
01234567
12345678
```

如果输入值不变，重点检查：

1. `SW13` 是否真的拨回 0；
2. 是否使用了 `SW[9:6]`，而不是 `SW[3:0]`；
3. `SW[5:0]` 是否保持 `000000`；
4. 汇编中是否通过以下代码取数：

```asm
srli  x7, x5, 6
andi  x7, x7, 15
```

### 8.4 第 3 关：确认 key 输入阶段

明文或密文输入完 8 位后，程序应显示：

```text
A0000000
```

这表示进入 16-bit key 输入阶段。

输入 key 例如 `2B7E` 时，显示应逐步变化：

```text
A0000002
A000002B
A00002B7
A0002B7E
```

如果无法进入 `A0000000`，说明 8 位输入计数没有结束，重点检查 `SW13` 的确认次数和松手机制。

### 8.5 第 4 关：确认算法输出

推荐固定测试向量：

```text
plaintext = 12345678
key       = 2B7E
```

加密模式最终应显示：

```text
36C8930C
```

解密模式输入：

```text
ciphertext = 36C8930C
key        = 2B7E
```

最终应显示：

```text
12345678
```

如果输入过程正确但最终结果不对，重点检查：

1. `x30 = 0x0000FFFF` 的掩码是否正确；
2. `derive_keys` 是否正确产生 `K0~K3`；
3. `encrypt32` 是否按 `K0,K1,K2,K3` 正序调用；
4. `decrypt32` 是否按 `K3,K2,K1,K0` 逆序调用；
5. `enc_round` 和 `dec_round` 是否使用了正确的 Feistel 公式。

### 8.6 第 5 关：利用调试通道观察 CPU 运行情况

当前 `SW[5:0]` 调试通道如下：

| `SW[5:0]` | 显示内容 | 用途 |
|---|---|---|
| `000000` | 应用输出 `disp_data` | 正常演示使用 |
| `000001` | `pc[31:2]` | 查看当前指令编号 |
| `000010` | `pc` | 查看 PC 是否进入结果死循环 |
| `000011` | `instr` | 查看当前指令 |
| `000100` | `cpu_data_addr` | 查看 CPU 当前数据地址 / ALU 结果 |
| `000101` | `cpu_data_out` | 查看 CPU 当前写数据 |
| `000110` | `dm_dout` | 查看数据 RAM 输出 |
| `000111` | `{ram_addr, 2'b00}` | 查看数据 RAM 字节地址 |
| `1xxxxx` | `reg_data` | 当前版本未接通，通常显示 0 |

注意：`cpu_data_addr` 在非访存指令周期也会显示 ALU 结果，因此它不总是有效 RAM 地址。

### 8.7 第 6 关：确认结果是否写入数据 RAM

加密模式完成后，结果区应为：

| 地址 | 内容 |
|---|---|
| `0x180` | plaintext |
| `0x184` | master key |
| `0x188` | ciphertext output |
| `0x194` | final result |
| `0x198` | K0 |
| `0x19C` | K1 |
| `0x1A0` | K2 |
| `0x1A4` | K3 |

解密模式完成后，结果区应为：

| 地址 | 内容 |
|---|---|
| `0x18C` | ciphertext input |
| `0x184` | master key |
| `0x190` | plaintext output |
| `0x194` | final result |
| `0x198` | K0 |
| `0x19C` | K1 |
| `0x1A0` | K2 |
| `0x1A4` | K3 |

如果怀疑内存写入有问题，可用 `SW[5:0]=000100/000101/000110/000111` 观察地址、写数据、RAM 输出和 RAM 地址。

## 10. 常见问题

### Q1：为什么 `SW[5]=1` 查看 x30 时显示 `00000000`？

因为当前版本的 `reg_data` 在 `RVCPUSOC_Top.v` 中被固定为 `32'b0`，寄存器堆没有调试读口。它不能证明 `x30` 真的为 0。

### Q2：如何确认 `x30` 真的等于 `0x0000FFFF`？

临时在 `li x30, 0x0000ffff` 后加入：

```asm
addi  x10, x30, 0
sw    x10, 0(x3)
```

若 `SW[5:0]=000000` 时显示 `0000FFFF`，则说明 `x30` 正确。

### Q3：为什么一定要用 `SW[9:6]` 输入数字？

因为 `SW[5:0]` 被 `Multi_CH32` 用作显示通道选择。如果使用 `SW[3:0]` 输入数字，会导致输入数字时显示通道也被改变。因此本程序使用 `SW[9:6]` 输入十六进制位，保持 `SW[5:0]=000000` 稳定显示应用输出。

### Q4：为什么切换加密/解密模式后没有变化？

`SW14` 只在程序启动/复位时读取一次。切换模式后需要重新复位 CPU。

### Q5：为什么输入时数字重复或跳位？

通常是 `SW13` 没有拨回 0，或者拨动过快。每确认一位都应执行：

```text
SW13: 0 -> 1 -> 0
```

必要时可以在 `input_hex_n` 的按下检测处加入短延时消抖。
